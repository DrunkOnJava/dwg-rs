# Clean-room policy

`dwg-rs` is developed clean-room from public specifications and
first-party byte inspection of publicly-available DWG files. This
document describes which sources are allowed and which are not, and
how contributors should certify compliance in pull requests.

It exists because the DWG ecosystem has a long history of licensing
disputes and multiple permissively-licensed efforts have been
complicated by upstream source contamination. Writing this policy
down makes the project's posture auditable by legal reviewers and
defensible against future claims.

## Allowed sources

Contributors may consult and cite any of the following while
developing `dwg-rs`:

- The **Open Design Alliance's freely-redistributable** *Open Design
  Specification for .dwg files* (v5.4.1 or later). This is a public
  PDF at
  [opendesign.com/files/guestdownloads/OpenDesign_Specification_for_.dwg_files.pdf](https://www.opendesign.com/files/guestdownloads/OpenDesign_Specification_for_.dwg_files.pdf).
- **Public academic papers** on CAD file formats, the LZ77 family,
  Reed-Solomon codes, and related algorithms.
- **Publicly-available DWG files** whose redistribution terms permit
  use for interoperability testing (e.g., the `nextgis/dwg_samples`
  repository on GitHub, or sample files released by organizations
  that allow third-party consumption).
- **Self-generated byte fixtures** written by hand to exercise
  specific decoder code paths.
- **Algorithm descriptions** in permissively-licensed projects
  (ACadSharp comments, LibDXF documentation, academic thesis work).
  Only the high-level algorithm description — never the implementing
  source code.
- **Standard Rust crates** on crates.io that are either Apache-2.0,
  MIT, or dual-licensed and compatible with Apache-2.0
  redistribution.
- **Rust language reference and standard-library docs**.
- **Hex editors, binary-diff tools, and disassemblers** applied to
  publicly-released DWG files — for **understanding format
  structure**, not for reverse-engineering Autodesk's or ODA's
  implementation binaries.

## Forbidden sources

Contributors must **not** consult, port, adapt, or quote from any of
the following while writing code for `dwg-rs`:

- **Autodesk SDK source code** — RealDWG, ObjectARX, ObjectDBX, or
  any other Autodesk-distributed SDK, whether or not the contributor
  has signed an NDA or developer agreement.
- **Open Design Alliance (ODA) SDK source** — "Drawings SDK" /
  "Teigha" / successor products. The ODA's *specification PDF* is
  explicitly allowed (see above); the ODA SDK source is explicitly
  forbidden.
- **LibreDWG** — GPL-3-licensed; source inspection would taint this
  crate's Apache-2.0 license.
- **Decompilations of Autodesk or ODA binaries** (AutoCAD,
  ObjectARX.dll, Teigha_*.dll, etc.).
- **Leaked or unofficial Autodesk / ODA documentation** that is not
  the public *Open Design Specification* PDF.
- **Any DWG implementation code whose license is GPL, LGPL, AGPL,
  SSPL, or proprietary** — including snippets found on Stack
  Overflow, GitHub Gists, or archived forum posts if the code
  lacks a permissive license.

## Contributor declaration

Every pull request that adds or modifies decoder / encoder logic
must include a declaration in the PR body. The PR template includes
a checkbox for this; the text of the declaration is:

> **Clean-room declaration.** I certify that:
>
> 1. All code and tests in this PR were written from scratch by me
>    or other individuals listed as co-authors.
> 2. I consulted only sources listed as allowed in `CLEANROOM.md`.
>    In particular, I did not consult Autodesk SDK source, ODA SDK
>    (Drawings SDK / Teigha) source, LibreDWG source, decompilations
>    of Autodesk or ODA binaries, or any copyleft-licensed DWG
>    implementation code.
> 3. Where this PR cites a reference (ODA spec section, algorithm
>    description, public paper), the citation is included in the
>    diff as a comment or doc link.
> 4. If I have ever had access to any of the forbidden sources in a
>    commercial or NDA'd context, I have disclosed that in the PR
>    discussion and the maintainer may mark this PR as needing
>    additional review.

A PR that cannot include this declaration truthfully will be
declined. Reviewers are asked to treat the declaration as load-
bearing: if something in the diff looks suspiciously close to a
known forbidden source, the PR is blocked until the author can
demonstrate clean-room provenance.

## For legal reviewers

If you are evaluating `dwg-rs` for commercial adoption and need to
audit the clean-room process end-to-end:

- Every merged PR passes through the declaration above. The
  `.github/PULL_REQUEST_TEMPLATE.md` file is the canonical form.
- `CONTRIBUTING.md` reiterates the rule from the contributor
  workflow angle.
- `NOTICE` summarizes the third-party authority (Sega v. Accolade,
  Sony v. Connectix, Baker v. Selden, etc.) that supports
  independent file-format reverse engineering for interoperability
  — this is context, not a legal opinion.
- `SECURITY.md` and `THREAT_MODEL.md` are separate documents
  governing vulnerability handling.

For direct questions about the clean-room posture, open a GitHub
Discussion or email the maintainer. For legal questions about
whether this project is safe for your specific use case, consult
your own counsel.
